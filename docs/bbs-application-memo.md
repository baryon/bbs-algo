# BBS 落地设计备忘录

副标题：围绕 AI Agent 自我迭代闭环、支付授权与自动化执行的工程化思考

## 1. 这份备忘录要回答什么问题

这份备忘录不再重复解释论文细节，而是聚焦一个更直接的问题：

`BBS 到底应该怎样落到真实系统里，尤其是带 AI Agent 的自动化闭环系统里？`

这里默认的背景是：

- 系统里有一个 Agent，它会规划、执行、调用工具、生成代码、发起支付、调用 API，或者做企业内部审批动作。
- 我们希望 Agent 能自主工作，但又不希望它拥有“拿到私钥就能做任何事”的无限权限。
- 我们希望某些硬约束不是靠提示词、代码约定或后端风控来保证，而是直接和签名能力本身绑定。

这正是 BBS 最值得思考的落地方向。

## 2. 我的总体判断

### 判断 1：BBS 最适合做“硬边界控制”，不适合做“质量评价”

BBS 的强项是：

- 限制支付金额
- 限制收款白名单
- 限制频率
- 限制允许调用的 API
- 限制允许执行的动作类型
- 限制资源上限

这些都属于 `可形式化、可验证、可编码进约束系统` 的硬规则。

BBS 不擅长的事情包括：

- 判断代码写得优不优雅
- 判断 Agent 的规划是否聪明
- 判断文案是否有说服力
- 判断某个设计方案是否“最好”

这些更像软评价问题，通常依赖语义判断、上下文解释和主观标准，不适合直接塞进 BBS。

### 判断 2：BBS 不是替代整个 Agent 系统，而是替代“无限签名权”

如果系统里没有强授权动作，BBS 不一定值得引入。  
如果系统里存在高风险动作，而这些动作目前靠普通密钥 + 软件规则约束，那么 BBS 的价值就很明显。

因此，BBS 更像是：

`Agent 系统里的密码学硬边界层`

而不是：

`Agent 系统的全部安全机制`

### 判断 3：默认不需要先上区块链

如果目标是控制一个 Agent 的支付、API 调用、代码部署或自动化执行权限，那么最自然的第一步不是做链，而是：

- 做一个本地或中心化的受限授权系统
- 让 Validator 只接受满足 BBS 约束的动作
- 把 BBS 放在高风险动作入口前

也就是说，BBS 先作为 `受限签名权系统` 落地，比先作为 `新型账本原语` 落地更现实。

## 3. 一个统一的闭环框架

我建议把所有场景都抽象成同一个闭环：

`Agent -> Action -> BBS Sign -> Validator -> Feedback -> Retry -> Execute`

下面分别解释。

### 3.1 Agent

Agent 负责理解任务、调用工具、生成候选结果，并决定是否发起某个受限动作。

这里的“动作”可以是：

- 发起支付
- 调用某个外部 API
- 提交部署请求
- 执行数据库写入
- 发送高权限审批消息

### 3.2 Action

所有高风险动作都必须先被表示成一个标准化对象，而不是一段自然语言。

例如支付 action：

```json
{
  "type": "payment",
  "amount": 168.50,
  "currency": "USD",
  "recipient": "vendor_123",
  "invoice_id": "inv_20260311_001",
  "reason": "design iteration payment",
  "epoch": 202603111030,
  "nonce": "n-001"
}
```

例如 API 调用 action：

```json
{
  "type": "api_call",
  "service": "payment-gateway",
  "endpoint": "/v1/payouts",
  "method": "POST",
  "body_hash": "....",
  "epoch": 202603111035,
  "nonce": "n-002"
}
```

这里的关键不是字段长什么样，而是：

- 字段要稳定
- 编码要规范
- 最终要能哈希绑定到签名和证明里

### 3.3 BBS Sign

Agent 拿到的不是普通私钥，而是 `带策略边界的签名能力`。

它对 action 生成：

- 一个签名对象
- 一个与策略约束绑定的证明

这个证明至少要说明两件事：

1. 这次 action 属于该身份允许的动作空间
2. 这次 action 满足预先承诺的硬规则

### 3.4 Validator

Validator 是系统中的硬门。

它不判断“Agent 做得好不好”，只判断：

- action 是否与签名绑定
- 证明是否有效
- 必要的外部上下文是否匹配
- 是否满足系统当前允许执行的条件

只有通过 Validator，动作才会进入真正执行层。

### 3.5 Feedback

若验证失败，系统不能只返回一个笼统的 `denied`，而应该返回结构化原因。

例如：

```json
{
  "status": "rejected",
  "reasons": [
    "amount_exceeds_limit",
    "recipient_not_in_whitelist"
  ]
}
```

这样 Agent 才能进行下一轮修正。

### 3.6 Retry

Agent 根据失败原因修正 action，再重新签名、再提交。

这一步非常重要，因为它把 BBS 从“静态授权工具”变成“反馈闭环中的控制器”。

这里的控制结构很像：

- 策略是目标边界
- Validator 是测量与判定环节
- Feedback 是误差返回
- Agent 是调节器

### 3.7 Execute

只有 Validator 通过，系统才真正执行动作。

这里必须坚持一个工程纪律：

`执行器不能接受未经 BBS 校验通过的高风险请求。`

如果执行器还有旁路接口，那 BBS 的硬边界就被架空了。

### 3.8 Agent 是否直接拿私钥，以及公钥由谁掌握

这个问题必须说清楚，否则工程实现很容易走偏。

首先，Agent `不一定直接持有原始私钥`。更准确的说法应当是：

`Agent 必须拥有某种可代表该身份发起 BBS 签名的能力。`

工程上我建议区分三种模式：

1. `Agent 直接持有受限私钥`
   - 最直观，实验系统最容易做
   - 但风险最高
   - 只适合原型验证，不适合作为默认生产方案

2. `Agent 不持有原始私钥，只调用受控 Signer`
   - 私钥放在独立 signer、HSM 或安全模块里
   - Agent 提交结构化 action
   - signer 在允许范围内生成 BBS 签名
   - 这是我更推荐的生产模式

3. `Agent 使用短生命周期委托身份`
   - 不给 Agent 长期根密钥
   - 只给它一个时间窗口有限、用途有限的受限身份
   - 更适合需要轮换、撤销和按任务授权的系统

这里还要强调一点：  
如果按 BBS 论文的思路，Agent 拿到的并不是“普通 ECDSA 私钥”那种对任何消息都能签的裸签名能力，而是：

- 某个 `BBS 身份` 的秘密签名材料
- 或者调用该 BBS 身份签名器的权限

也就是说，系统设计上不应该让 Agent 拿到“万能钥匙”，而应该让它拿到“带边界的签名身份”。

对应地，Validator 必须掌握以下至少一项：

- 某个 Agent 对应的唯一公钥
- 一组允许公钥白名单
- 身份注册表或证书链
- `agent_id -> allowed_pk` 的绑定关系

Validator 的最小身份检查包括：

1. 这个公钥是否属于系统认可身份
2. 这个公钥是否允许执行当前这类 action
3. 这次签名和证明是否确实由该公钥对应身份产生

因此，一个非常直接的工程规则是：

`如果请求来自不在允许集合里的公钥，Validator 应直接拒绝。`

但还要再补一句：

`公钥正确并不代表请求一定通过。`

即使请求来自已注册公钥，只要：

- proof 无效
- action 与签名绑定不一致
- action 超出该身份允许的策略边界

Validator 一样必须拒绝。

### 3.9 为什么要零知识验证，以及能否换成别的签名方式

这要看你到底想要多强的保证。

如果目标只是：

- 确认“是谁发起的请求”
- 然后由软件层重新检查规则

那么普通签名就够了。

如果目标是：

`让“动作满足策略”本身成为签名有效性的组成部分`

那么零知识证明就不是点缀，而是核心部件。

零知识在这里承担两个作用：

1. `证明合规`
   - 证明这次 action 满足额度、白名单、频率等硬约束

2. `隐藏不该公开的内容`
   - 不必向 Validator 明文暴露所有内部参数、私密 witness 或敏感业务细节

换句话说，Validator 验证的不是“一个普通签名”，而是：

`一个已经把合规性证明打包进去的可验证动作。`

当然，工程上也有其他替代方案，但安全边界不同。

`方案 A：普通签名 + 策略引擎`

- Agent 用普通签名签 action
- Validator 收到后在软件里跑规则
- 不符合就拒绝

优点：

- 最容易实现
- 性能最好
- 和现有系统兼容性最高

缺点：

- 规则 enforcement 仍然在软件层
- 它不能从数学上限制“持钥者自己乱签”
- 如果策略引擎有 bug、被绕过或有旁路，风险仍在

`方案 B：HSM / 受控 Signer + 普通签名`

- Agent 不持有原始私钥
- 私钥在 HSM 或 signer 服务中
- signer 在签名前检查规则，通过才签

优点：

- 比 Agent 直接持钥安全很多
- 工程上比 BBS 更现实
- 很适合作为第一阶段落地方案

缺点：

- 本质上仍然依赖 signer 软件或设备逻辑正确
- 它是“受控签名服务”，不是“策略已经写进签名有效性本身”

`方案 C：BBS + 零知识验证`

- Agent 或受控 signer 生成带策略证明的签名对象
- Validator 验证 action 绑定和 proof

优点：

- 最接近“硬边界控制”
- 更符合论文想要的“软件不可覆盖”方向
- 更适合需要隐私和更强密码学约束的系统

缺点：

- 工程复杂度最高
- 证明系统和调试成本都更高
- 对接收方能力要求更高

所以，更实际的结论不是“只能用 BBS + ZK”，而是：

- 如果你要最快落地，用 `受控 signer/HSM + 普通签名 + Validator`
- 如果你要最强的密码学边界，并愿意承担复杂度，再考虑 `BBS + ZK`

一句话总结：

`零知识的必要性，不在于它更高级，而在于你是否真的要把“合规性证明”做成签名有效性的一部分。`

## 4. 为什么这个闭环有意义

它的意义不在于“Agent 变聪明了”，而在于：

`即使 Agent 失控，它能造成的动作空间也被压缩到了一个预设边界之内。`

这会带来几个直接好处：

- 把高风险动作从“先执行再补救”改成“先证明再执行”
- 把大量软件层策略转成签名能力层策略
- 让 Agent 能在较大自由度下自主迭代，但出不了硬边界
- 让 Validator 的职责清晰，不必理解全部业务语义

## 5. 支付场景详解

这里用一个最具体的例子来展开：

`规则：200 USD 以内的白名单用户`

### 5.1 最小规则集

支付系统中的最小硬规则可以先设成：

- `amount <= 200 USD`
- `recipient in whitelist`
- `currency == USD`
- `type == payment`
- `nonce` 未被重放
- 可选：单位时间内支付次数不超过 `N`
- 可选：每日累计金额不超过 `M`

### 5.2 Agent 的工作流程

Agent 在完成某项工作后，可能生成一个支付请求，例如给外部设计师或供应商付款。

这个时候它不能直接调用支付接口，而要先：

1. 生成 payment action
2. 对 action 做规范化编码
3. 使用 BBS 生成受限签名
4. 提交给 Validator

### 5.3 Validator 的职责

Validator 至少要做下面几类检查：

1. `payload binding`
   - 实际提交的 payment action 必须与签名/证明绑定的 action 完全一致。

2. `policy proof verification`
   - 验证这笔支付满足额度、白名单和其他硬约束。

3. `replay protection`
   - 防止同一个 action 或 nonce 被重复执行。

4. `execution gate`
   - 只有通过验证的请求，才能被真正转发到支付执行器。

### 5.4 一个合理的失败反馈

支付 Validator 可以返回类似这样的反馈：

```json
{
  "status": "rejected",
  "reasons": [
    {
      "code": "amount_exceeds_limit",
      "detail": "requested 243.00 USD, max 200.00 USD"
    },
    {
      "code": "recipient_not_whitelisted",
      "detail": "recipient vendor_999 is not approved"
    }
  ]
}
```

这样 Agent 可以：

- 降低付款金额
- 更换为白名单收款方
- 或请求人工升级权限

### 5.5 这个场景里 BBS 的真实价值

支付场景中，BBS 的价值不是“支付流程更好看”，而是：

- Agent 拿到了签名能力，但没有无限支付能力
- 即使 Agent 被 prompt injection、bug 或工具链污染，它也无法提交越界支付
- 风险控制点从“支付前的软件检查”前移成“签名时的数学限制”

### 5.6 还不够的地方

支付场景里，BBS 仍然不能自动解决这些问题：

- 白名单本身配错了
- 发票是伪造的
- 业务理由是假的
- 链下收款身份和链上地址映射错了
- 币种换算逻辑有误

也就是说，BBS 保的是 `动作边界`，不保 `业务事实真伪`。

## 6. 代码生成与自动化开发

这个场景要更谨慎，因为“代码质量”很难写进 BBS。

### 6.1 适合用 BBS 约束的部分

如果 Agent 可以自动生成代码、运行测试、发起部署，那么 BBS 适合约束的是：

- 只能写入指定仓库或目录
- 只能修改白名单文件类型
- 不能直接改生产环境密钥配置
- 只能触发指定的 CI/CD 接口
- 部署只能到 staging，不能直接到 production
- 单次资源消耗在上限内

这些都是明确的动作边界。

### 6.1.1 危险 action 约束其实是这一场景里最适合 BBS 的部分

代码生成场景里，最值得优先用 BBS 管的，不是“代码好不好”，而是“Agent 能不能做出危险操作”。

典型例子包括：

- 删除数据库
- 删除系统目录文件
- 修改生产环境配置
- 对生产库执行 `DROP`、`TRUNCATE`、大范围 `DELETE`
- 在错误环境中运行迁移
- 执行高风险 shell 命令

这些操作有一个共同点：

`它们都可以被表达成结构化 action，并且可以被明确限制。`

例如，系统可以把危险动作统一建模为：

- `fs.delete(path)`
- `fs.write(path, content_hash)`
- `db.query(query_type, target, scope)`
- `db.migrate(env, migration_id)`
- `shell.exec(command_id, args, cwd)`
- `deploy.start(env, artifact_id)`

一旦 action 被结构化，就可以很自然地加边界，例如：

- `path` 必须在 `/workspace/sandbox/**`
- 禁止删除 `/etc`、`/usr`、`/bin`、系统配置目录和宿主挂载目录
- `env != production`
- `query_type != DROP`
- `table` 不属于核心表
- `command_id` 必须在白名单内
- 每次删除的文件数、每次改动的目录范围、每次迁移的目标环境都有限制

从 BBS 适配性的角度，这类约束比“代码质量评价”更接近它的理想用法，因为它们都是：

- 可形式化
- 可枚举
- 可验证
- 能明确拒绝

### 6.1.2 但前提是执行入口必须被收口

这里有一个不能回避的工程前提：

`Agent 不能直接拿到一个能删库删系统文件的原始 shell 或高权限数据库连接。`

否则即使你设计了再漂亮的 BBS 规则，Agent 也可以绕开结构化 action 通道直接执行危险操作。

所以一个靠谱的工程架构应该是：

- Agent 只能提交结构化 action
- BBS/Validator 只对结构化 action 放行
- 真正的文件系统、数据库、部署系统操作由受控执行器完成
- 执行器不接受未经校验的原始请求

换句话说，BBS 约束的不是“任意自然语言行为”，而是：

`所有真正能落到系统里的高风险执行入口。`

### 6.1.3 BBS 不是唯一防线

即使在这类危险 action 场景里，我也不建议把所有安全性都押在 BBS 上。

更稳的做法是多层防护同时存在：

- `BBS`
  - 约束允许发起什么动作

- `操作系统/容器沙箱`
  - 限制进程实际能访问什么目录、设备和系统调用

- `数据库最小权限账号`
  - 限制 SQL 真正能做什么

- `环境分层`
  - 默认只允许对沙箱、测试库、staging 进行操作

- `审计与回滚机制`
  - 在约束失效或配置错误时提供补救路径

因此，对开发 Agent 来说，更准确的说法不是：

`BBS 可以单独阻止删库删系统文件。`

而是：

`BBS 非常适合成为危险 action 的第一道硬边界，但它必须和沙箱、最小权限、受控执行器一起工作。`

### 6.2 不适合用 BBS 约束的部分

BBS 不适合直接判断：

- 代码是否优雅
- 架构是否合理
- 测试是否覆盖充分到“足够好”
- 需求理解是否正确

这些更适合交给：

- 单元测试
- 静态分析
- Code review agent
- 人工审批

### 6.3 这一场景的合理闭环

我建议拆成三层：

1. `BBS`
   - 约束 Agent 可以做什么操作

2. `Deterministic Validator`
   - 运行测试、lint、schema 校验、构建检查

3. `Soft Evaluator`
   - 判断代码是否真正达到预期质量

一个更现实的流程是：

`Agent 生成补丁 -> BBS 签署“允许的修改动作” -> 测试器验证 -> 若失败则反馈 -> Agent 修补 -> 通过后提交`

### 6.4 一个最小落地版本

先不要让 Agent 拥有“修改任何仓库并部署”的权力。更好的起点是：

- 只允许写某个沙箱目录
- 只允许触发测试
- 不允许直接推送 production
- 不允许修改 secrets

这是一个非常适合 BBS 的低风险切入点。

## 7. API 调用授权

这是我认为除支付之外，最自然的第二落地方向。

### 7.1 为什么这个方向自然

很多高风险动作本质上都可以归结为 API 调用：

- 发起转账
- 写入数据库
- 删除资源
- 触发发布
- 调用客户数据接口

因此，如果把“可调用哪些 API、什么参数范围内可调用”绑定到 BBS，很多问题会变得清楚。

### 7.2 适合绑定的规则

- 只允许调用某个服务
- 只允许调用白名单 endpoint
- 只允许某些 method
- 参数大小在限制范围内
- 目标账户或资源 ID 必须在允许集合中
- 调用频率不能超限
- 每日累计调用预算不能超限

### 7.3 最小设计

一个 API action 可以包含：

```json
{
  "type": "api_call",
  "service": "crm",
  "endpoint": "/v1/customer/update",
  "method": "POST",
  "resource_id": "customer_123",
  "body_hash": "....",
  "epoch": 202603111120,
  "nonce": "n-101"
}
```

然后由网关只接受通过 BBS Validator 的请求。

### 7.4 风险

这个模型依然不能保证：

- 请求内容在业务意义上“合理”
- API 本身没有后门
- 被允许的接口不会造成组合型风险

因此，BBS 适合当“第一道硬边界”，不是唯一防线。

## 8. 企业审批与财务机器人

这一类系统和支付例子非常接近，但更强调流程链路。

### 8.1 适合放进 BBS 的规则

- 单笔审批金额上限
- 只能审批指定类别
- 只能面向白名单供应商
- 某个周期内总额度有限
- 某个角色只能发起、不能终审
- 某类文档缺失时不能提交

### 8.2 一个合理的架构

可以把企业审批系统拆成：

- `Planner Agent`
  - 负责准备审批材料

- `BBS Signer`
  - 对“审批动作”生成受限签名

- `Approval Validator`
  - 验证动作是否满足角色和额度规则

- `Workflow Engine`
  - 真正推进流程状态

### 8.3 为什么这里有价值

企业系统里最常见的问题不是密码学伪造，而是：

- 自动化脚本拿到高权限账号
- 规则写在后端里，容易被改
- 流程系统里存在隐藏旁路

BBS 可以把“谁能审批到什么程度”前移成密钥能力边界。

### 8.4 局限

它仍然无法替代：

- 真实发票核验
- 线下合同真实性
- 业务合理性判断
- 组织责任划分

## 9. 自动化执行、机器人和 IoT

这是一个经常被忽视、但其实很适合 BBS 的方向。

### 9.1 为什么适合

机器人和设备的许多危险动作都天然有“幅度、对象、频率、地理区域”这些可形式化边界。

例如：

- 机械臂移动范围不能超出安全区
- 无人机不能飞出地理围栏
- 工业设备不能在 1 分钟内重复执行超过 N 次高风险动作
- IoT 设备只能向白名单控制器上报或接收命令

### 9.2 一种思路

把设备控制命令也变成 action，并要求所有高风险命令都带 BBS 签名。

执行器只接受“满足边界约束的命令”。

### 9.3 这里最重要的提醒

如果执行器还有一个不经过 BBS 的调试接口，那整个模型就会被绕过。  
所以设备场景对“旁路消除”要求更高。

## 10. 是否需要区块链

这是一个必须单独讨论的问题。

### 10.1 我的判断：大多数第一阶段落地不需要区块链

如果目标是：

- 限制 Agent 支付
- 限制高风险 API 调用
- 限制部署权限
- 限制企业自动审批

那么一个中心化或半中心化的 Validator 就足够了。

这时候系统需要的是：

- 受限签名能力
- 严格的执行入口
- 反馈循环
- 日志与审计

并不需要先引入：

- 共识协议
- 公开账本
- 去中心化排序
- 代币经济

### 10.2 什么时候区块链才有意义

只有在下面几类条件出现时，区块链或共享账本才更值得考虑：

- 有多个互不完全信任的参与方
- 审计必须是公共可验证的
- 需要跨组织共享动作历史
- 需要去中心化排序和结算
- 需要把权限边界和资产结算结合起来

例如：

- 多机构共同管理自动化支付网络
- 跨组织 API 代理市场
- 公共可审计的 Agent 执行账本
- 需要链上结算的自治软件系统

### 10.3 即便上链，也不代表一定需要传统 gas

如果系统采用论文设想的思路：

- 不做通用 VM 执行
- 主要做证明验证
- 用其他机制限制 spam

那么传统意义上的 gas 可以弱化甚至被替换。

但这不等于系统没有成本。成本仍然存在于：

- 证明生成
- 证明验证
- 存储
- 带宽
- 排序
- 节点激励

所以更准确的说法应当是：

`BBS 有可能减少或替代传统 gas 机制，但不会让资源成本消失。`

### 10.4 我的建议：先不上链，先闭环

如果今天真的要做一个 BBS 落地系统，我会建议按下面顺序推进：

1. 先做单组织内的 BBS Agent 闭环
2. 先解决 action 建模、签名绑定、Validator、反馈协议
3. 先在支付或 API 调用这类边界清楚的动作上试点
4. 只有在确实需要多方共享账本时，再讨论是否上链

这条路径比“先造一个 BBS 链”现实得多。

## 11. BBS 在闭环里到底应该签什么

这是一个工程上很关键的问题。

### 11.1 只签动作，不签自然语言意图

我不建议 BBS 直接签“任务描述”或“自然语言计划”，因为这类内容语义模糊，验证稳定性差。

我建议签的是：

- 结构化 action
- action 的规范化编码
- 必要的上下文摘要

### 11.2 在支付例子里，最少要签哪些内容

至少包括：

- 动作类型
- 金额
- 币种
- 收款方标识
- 发票或订单摘要
- 时间窗口字段
- nonce

如果只签金额、不签收款方，那么系统仍然可能被“换目标对象”绕过。

### 11.3 在代码生成例子里，最少要签哪些内容

至少包括：

- 仓库标识
- 允许修改的目录范围
- 补丁摘要或提交摘要
- 目标环境
- 可调用的部署接口
- 执行窗口

### 11.4 一个原则

`BBS 必须签“真正要执行的对象”，而不是只签一个模糊许可。`

## 12. Validator 的最小职责

一个靠谱的 Validator 至少要有下面这些职责：

1. 验证 action 与签名绑定
2. 验证 BBS 证明本身
3. 检查 replay / nonce / 时间窗
4. 把失败原因结构化返回
5. 保证执行器只吃经过验证的请求
6. 记录审计日志

如果 Validator 做不到第 5 条，整个 BBS 方案都不牢。

## 13. 策略升级、撤销与例外处理

这部分很现实，因为很多规则不是永远不变的。

### 13.1 策略升级

如果策略写死在 KeyGen 或某个长期身份里，那么升级会比较重。  
所以工程上可以考虑：

- 使用短生命周期身份
- 对不同权限等级发行不同身份
- 用上层策略版本管理去替代频繁重建整个系统

### 13.2 撤销

如果 Agent 被怀疑失控，需要立即撤销或冻结其受限身份。

因此，实际系统仍然需要：

- 吊销列表
- 过期时间
- 紧急停用机制

### 13.3 例外处理

现实系统总会遇到超额但合理的情况。  
我建议不要试图在一个身份里把所有例外都编码进去，而是：

- 普通动作走普通受限身份
- 高权限例外走人工审批或更高权限身份

这样边界更清楚。

## 14. 风险与边界

这里列出最重要的边界，不然很容易把 BBS 想得过强。

### 风险 1：规则形式化不完整

如果你把错误的规则编码得非常严格，系统只会非常稳定地执行错误规则。

### 风险 2：旁路接口

如果执行器存在不经过 Validator 的后门、管理员接口或旧接口，BBS 会被架空。

### 风险 3：把软问题硬塞进 BBS

代码质量、业务合理性、战略判断这类问题，不应该强行塞进 BBS。  
那样只会让系统复杂度暴涨，收益却很小。

### 风险 4：性能与开发复杂度

引入 BBS 一定会提高：

- 开发复杂度
- 证明系统复杂度
- 运维复杂度
- 调试难度

所以它只适合高价值、高风险动作。

### 风险 5：生态兼容

绝大多数现有系统只会验证普通签名，不会理解 BBS。  
因此，第一阶段落地大概率要在你可控的执行入口内完成。

## 15. 我建议的最小落地路线

如果要做一个真正可用的 BBS Agent 系统，我建议按下面顺序推进。

### 第一步：选一个边界清楚的动作

优先级建议：

1. 支付
2. API 调用
3. 自动化部署
4. 企业审批

### 第二步：只做硬规则

不要一开始就追求“全自动安全 Agent”。  
先把最清晰的规则放进去：

- 金额上限
- 白名单
- endpoint 白名单
- 频率限制
- 环境限制

### 第三步：先做中心化 Validator

先保证闭环成立：

- action 标准化
- BBS 绑定
- 失败反馈
- 执行门禁

### 第四步：再讨论多方共享历史

如果之后真的需要：

- 跨组织协作
- 公开审计
- 去中心化结算

再考虑账本或链。

## 16. 从控制论视角看 BBS 闭环

前面描述的闭环框架 `Agent -> Action -> BBS Sign -> Validator -> Feedback -> Retry -> Execute`，如果从控制论（Cybernetics）的角度重新审视，会发现它不仅仅是一个安全机制，而是一个完整的负反馈控制系统。

### 16.1 经典控制回路的映射

控制论中一个标准的闭环控制系统包含以下要素：

| 控制论概念 | BBS 闭环对应 |
|-----------|-------------|
| 参考信号（Reference） | 策略约束 + 任务目标 |
| 控制器（Controller） | Agent |
| 执行器（Actuator） | BBS Signer + 执行服务 |
| 被控对象（Plant） | 外部系统（支付网关、API、部署环境等） |
| 传感器（Sensor） | Validator |
| 误差信号（Error Signal） | 结构化拒绝反馈 |

画成图是这样的：

```
                    ┌──────────────────────────────────────────────────┐
                    │                                                  │
                    ▼                                                  │
  ┌──────────┐   action   ┌──────────┐  signed   ┌──────────────┐    │
  │  Agent   │ ─────────→ │  BBS     │ ───────→  │  Validator   │    │
  │(控制器)  │            │  Signer  │  request  │  (传感器)     │    │
  └──────────┘            └──────────┘           └──────┬───────┘    │
       ▲                                                │            │
       │                                          pass? │            │
       │                                         ┌──────┴──────┐    │
       │                                         │             │    │
       │    ┌────────────────────────┐      Yes  ▼        No   ▼    │
       │    │     Execute            │◀──────────┘    ┌────────┐    │
       │    │   (被控对象)           │                │Feedback│    │
       │    └────────────────────────┘                │(误差)  │────┘
       │                                              └────────┘
       │                                                   │
       └───────────────────────────────────────────────────┘
                        结构化反馈驱动下一轮迭代
```

在传统控制系统中，控制器根据参考信号与实际输出之间的误差来调整输入。在 BBS 闭环中，这个过程完全一样：

1. **参考信号**：策略定义了"什么是允许的"
2. **Agent 输出**：生成一个候选 action
3. **Validator 测量**：检验 action 是否满足参考信号
4. **误差计算**：如果不满足，生成结构化偏差描述
5. **反馈调节**：Agent 根据偏差信息修正 action
6. **再次提交**：新 action 进入下一个控制周期

### 16.2 签名验证天然就是反馈机制

这是一个容易被忽视的观察：

`BBS 的签名验证过程本身就是一个信息丰富的反馈通道。`

在传统密码学系统中，签名验证的结果通常是布尔值——通过或不通过。但在 BBS 闭环中，Validator 返回的不只是"拒绝"，而是结构化的拒绝原因：

```json
{
  "status": "rejected",
  "reasons": [
    {"code": "amount_exceeds_limit", "detail": "requested 243.00, max 200.00"},
    {"code": "recipient_not_whitelisted", "detail": "vendor_999 not in approved list"}
  ]
}
```

这意味着：

- **每一次验证失败都携带了梯度信息**：Agent 不仅知道"错了"，还知道"错在哪里"、"偏了多少"
- **反馈是可操作的**：Agent 可以直接根据 `amount_exceeds_limit` 降低金额，根据 `recipient_not_whitelisted` 更换收款方
- **收敛是可预期的**：只要 Agent 有基本的推理能力，每一轮反馈都应该缩小偏差

这比传统的"一刀切拒绝"好太多。传统系统的失败通常是：

```
403 Forbidden
```

Agent 能从这个里面学到什么？几乎什么都学不到。

### 16.3 关键洞察：任务目标也可以成为约束签名

前面讨论的所有场景，策略约束都是"安全边界"——金额上限、白名单、环境限制。这些都是"不能做什么"的约束。

但控制论的参考信号不一定只是"边界"，它也可以是"目标"。

这引出了一个重要的扩展方向：

`如果把 Agent 的任务目标也编码成验收条件，那么同一套 BBS 闭环就不仅能约束安全边界，还能驱动任务收敛。`

具体来说：

**安全约束型（原有用法）**

- 参考信号 = "金额 <= 200 USD，收款方在白名单内"
- Validator 验证的是"动作是否在允许范围内"
- 反馈驱动 Agent 把动作调整到边界以内

**任务目标型（扩展用法）**

- 参考信号 = "代码必须通过全部单元测试"、"API 响应时间 < 200ms"、"部署到 staging 后健康检查通过"
- Validator 验证的是"工作成果是否满足验收标准"
- 反馈驱动 Agent 迭代工作成果直到达标

两者在控制论结构上是同构的：

| | 安全约束模式 | 任务目标模式 |
|---|---|---|
| 参考信号 | 策略边界（不等式约束） | 验收标准（等式/质量约束） |
| Agent 输出 | 候选动作 | 工作成果 |
| Validator | 检查动作是否越界 | 检查成果是否达标 |
| 反馈 | "哪些规则被违反" | "哪些验收条件未通过" |
| 收敛目标 | 动作落入允许空间 | 成果满足验收标准 |

### 16.4 任务目标验收的具体机制

在任务目标模式下，闭环的运作方式如下：

**第一步：定义验收标准并绑定为约束**

```json
{
  "task_id": "task_20260311_001",
  "type": "code_generation",
  "acceptance_criteria": [
    {"criterion": "unit_tests_pass", "target": "all"},
    {"criterion": "lint_clean", "target": true},
    {"criterion": "type_check_pass", "target": true},
    {"criterion": "no_new_vulnerabilities", "target": true}
  ],
  "constraints": {
    "allowed_dirs": ["/workspace/src/**"],
    "max_files_modified": 10,
    "target_env": "staging"
  }
}
```

这里面同时包含了两类参考信号：
- `constraints`：传统安全边界（与之前一样）
- `acceptance_criteria`：任务目标验收条件

**第二步：Agent 完成工作后提交验收请求**

Agent 生成代码补丁后，不是直接提交，而是构造一个包含工作成果摘要的 action：

```json
{
  "type": "task_submission",
  "task_id": "task_20260311_001",
  "artifact_hash": "sha256:abc123...",
  "files_modified": ["src/auth.py", "src/auth_test.py"],
  "test_results_hash": "sha256:def456...",
  "agent_id": "dev_agent_01",
  "epoch": 202603111430,
  "nonce": "n-301"
}
```

**第三步：Validator 执行验收**

验收器（这里的 Validator 角色由一个确定性检验器承担）执行：

1. 运行单元测试 → 检查是否全部通过
2. 运行 lint → 检查是否清洁
3. 运行类型检查 → 检查是否通过
4. 运行安全扫描 → 检查是否有新增漏洞
5. 检查修改范围 → 是否在允许目录内
6. 检查文件数量 → 是否超限

**第四步：不通过则返回结构化反馈**

```json
{
  "status": "rejected",
  "task_id": "task_20260311_001",
  "reasons": [
    {
      "code": "unit_tests_failed",
      "detail": "3 of 47 tests failed",
      "failing_tests": ["test_token_refresh", "test_session_expire", "test_concurrent_login"]
    },
    {
      "code": "lint_violation",
      "detail": "2 warnings in src/auth.py:45, src/auth.py:78"
    }
  ]
}
```

**第五步：Agent 迭代修正**

Agent 根据反馈：
- 修复 3 个失败的测试用例
- 处理 2 个 lint 警告
- 生成新的工作成果
- 再次提交验收请求

**第六步：验收通过，放行执行**

```json
{
  "status": "accepted",
  "task_id": "task_20260311_001",
  "signature": "...",
  "proof": "..."
}
```

只有此时，补丁才被允许合入或部署。

### 16.5 控制论意义上的稳定性分析

从控制论角度看，这个闭环能否稳定收敛取决于几个条件。

我们在 `src/python/bbs_cybernetics_mvp.py` 中用可运行的代码验证了这些条件。这个 MVP 定义了三种反馈模式（`rich`、`coarse`、`none`）和四个场景，直接演示了反馈精度、Agent 执行器极限和目标可达性如何决定闭环行为。

**条件 1：反馈必须是信息充分的**

如果 Validator 只返回"不通过"，Agent 就像一个没有传感器精度的控制器——它只能盲目尝试。这正是为什么结构化反馈如此重要。

控制论 MVP 定义了三种反馈模式来验证这一点：

| 反馈模式 | Validator 返回内容 | Agent 修正策略 | 收敛效果 |
|---------|-------------------|---------------|---------|
| `rich`（精确反馈） | 违反原因 + 每个维度的精确偏差值 | 按偏差的 60% 做比例修正 | 快速收敛（3-4 轮） |
| `coarse`（粗反馈） | 仅违反原因，无偏差值 | 按固定步长盲目修正 | 收敛缓慢，可能在有限轮次内失败 |
| `none`（无反馈） | 无任何信息 | 无法修正 | 立即停止（`missing_feedback`） |

MVP 场景 `converges_with_structured_feedback` 演示了 rich 模式下 Agent 如何在几轮内收敛；`fails_with_coarse_feedback` 演示了相同目标、相同初始状态下，粗反馈导致收敛失败；`stops_without_feedback_channel` 演示了无反馈时 Agent 无法做任何修正。

这三个场景直观地证明了：**反馈精度直接决定收敛速度和成败。**

对应到控制论：
- `rich` = 高分辨率传感器，误差信号包含方向和幅度
- `coarse` = 低分辨率传感器，误差信号只包含方向
- `none` = 传感器断开，控制回路开环

**条件 2：Agent 的调节能力必须足够（执行器极限）**

控制论中，控制器必须有足够的"调节带宽"来消除误差。但同样重要的是，**执行器本身有物理极限**。

在 MVP 中，`AdaptiveAgent` 定义了明确的执行器边界：

- `max_quality_score = 90`：Agent 能达到的最高质量分
- `min_cost = 60`：Agent 能压到的最低成本
- `min_latency_ms = 120`：Agent 能优化到的最低延迟
- `min_risk_score = 8`：Agent 能降到的最低风险分

这意味着即使反馈再精确，如果策略要求 `quality >= 95` 而 Agent 最高只能做到 90，系统就永远无法收敛。MVP 场景 `fails_with_unreachable_target` 正是演示了这种情况——Agent 在每轮都在修正，偏差在缩小，但最终触碰到自身极限后停止，返回 `actuator_limits_reached`。

这引出了一个重要的工程原则：

`策略制定者必须了解 Agent 的实际能力边界，否则会设定不可达的目标。`

从控制论角度，这对应"执行器饱和"（Actuator Saturation）——当控制信号超出执行器的物理范围时，系统输出不再跟随参考信号。

**条件 3：参考信号必须是可满足的**

如果验收标准相互矛盾，或者在 Agent 执行器极限下不可能同时满足，系统就会陷入无效循环。这在控制论中对应"不可达的设定点"。

条件 2 中的执行器极限是一种情况；另一种情况是多个约束维度之间存在固有冲突——例如同时要求极低成本和极高质量，而 Agent 的能力空间中这两个维度负相关。

工程上应对这种情况的方式是：

- 设置最大重试次数（MVP 中通过 `max_iterations` 参数控制）
- 当 Agent 报告 `actuator_limits_reached` 时，升级为人工介入
- 让 Agent 能主动报告"当前约束下无法满足目标"，而不是无限循环

**条件 4：系统不能有正反馈回路**

如果 Agent 的修正反而让情况变差（例如修一个 bug 引入两个新 bug），系统就会发散。这在工程上需要通过以下方式缓解：

- 每次提交前运行全量回归测试，而不仅仅是修复的部分
- 保持修正的原子性，避免大范围连锁变更
- 监控偏差是否在逐轮减小，如果增大则触发告警

### 16.6 这个视角带来的认知升级

从控制论角度看 BBS 闭环，最大的认知升级是：

**升级 1：BBS 不仅是安全锁，它是控制回路的核心组件**

传统理解：BBS 是一把"限制 Agent 做坏事的锁"。
控制论理解：BBS 是一个"驱动 Agent 做对事的反馈环"。

两者并不矛盾，而是同一机制的两面：
- 安全约束 = 控制系统的饱和限幅器（Saturation Limiter），防止输出超限
- 任务验收 = 控制系统的参考跟踪（Reference Tracking），驱动输出趋近目标

**升级 2：签名不只是授权凭证，它是收敛证据**

每一次成功的签名验证，都是对"当前 action 满足全部约束"的数学证明。从控制论角度：

- 签名验证通过 = 系统输出落入目标域
- 签名验证失败 = 系统输出偏离目标域，误差信号触发修正

这意味着：一个 Agent 的签名历史，本质上是它的控制轨迹——记录了它如何从初始偏差逐步收敛到目标状态。

**升级 3：Validator 不只是看门人，它是测量装置**

在传统架构中，Validator 的角色是"允许或拒绝"。从控制论角度，Validator 的真正角色是"测量与反馈"——它是系统中的传感器，负责把"当前状态与目标状态之间的偏差"量化并返回给控制器。

这意味着 Validator 的设计重点不仅是"验证正确性"，还应该是"反馈信息量最大化"。一个只返回 `true/false` 的 Validator 是一个低分辨率传感器；一个返回结构化偏差描述的 Validator 是一个高分辨率传感器。

### 16.7 与传统 Agent 工作流的本质区别

现有的大多数 Agent 工作流也有"重试"机制，但它们和 BBS 闭环有本质区别：

| | 传统 Agent 重试 | BBS 控制闭环 |
|---|---|---|
| 反馈来源 | 异常捕获、日志解析、LLM 自省 | 密码学验证 + 确定性检查器 |
| 反馈可靠性 | 不确定，依赖 Agent 自身判断 | 确定性，不依赖 Agent 理解 |
| 通过标准 | Agent 自己认为"完成了" | Validator 独立验证"确实完成了" |
| 可绕过性 | Agent 可以"说服自己"跳过检查 | 签名机制不可自欺 |
| 收敛保证 | 无（可能无限循环或提前退出） | 有明确终止条件和升级机制 |

传统 Agent 重试的根本问题在于：`判断"是否完成"的权力掌握在 Agent 自己手里。`

这就像一个控制系统中，控制器同时充当传感器——它永远可以告诉自己"已经到位了"，即使实际上偏差很大。

BBS 闭环把"测量"从 Agent 中分离出来，交给一个独立的、不可被 Agent 操纵的 Validator。这在控制论中叫做 `分离原则`（Separation Principle）：控制与观测必须独立。

### 16.8 一个完整的例子：Agent 自主完成编码任务

把安全约束和任务目标结合在一起，一个完整的闭环过程如下：

```
任务：为系统添加 token 刷新功能

约束签名（安全边界）：
  - 只能修改 /workspace/src/auth/** 目录
  - 不能修改 /workspace/config/secrets/**
  - 不能直接访问生产数据库
  - 单次最多修改 15 个文件

验收签名（任务目标）：
  - 所有现有测试必须通过
  - 新增测试覆盖 token 刷新逻辑
  - lint 和类型检查必须通过
  - 无新增安全漏洞

循环 1：
  Agent 生成代码 → 提交
  Validator：unit_tests_failed (test_session_expire)，lint_violation (2 warnings)
  反馈 → Agent 修正

循环 2：
  Agent 修复测试 + lint → 提交
  Validator：unit_tests_pass，lint_clean，但 type_check_failed (auth.py:23 type error)
  反馈 → Agent 修正

循环 3：
  Agent 修复类型错误 → 提交
  Validator：全部验收条件通过 ✅
  签名验证通过 → 允许合入
```

在这个过程中：
- 安全约束确保 Agent 无论怎么迭代，都不会碰到不该碰的东西
- 任务目标确保 Agent 的工作成果达到预定义的验收标准
- 每一轮 Validator 反馈都在缩小偏差
- 最终的签名验证通过，是对"安全合规且任务完成"的双重数学证明

### 16.9 小结

从控制论视角看 BBS 闭环，核心结论是：

1. **BBS 闭环本质上是一个负反馈控制系统**，其中 Agent 是控制器，Validator 是传感器，策略和目标是参考信号
2. **签名验证天然携带反馈信息**，结构化的拒绝原因就是误差信号，驱动 Agent 迭代修正
3. **任务目标和安全约束在控制论框架下是同构的**，都是参考信号的不同形式——前者是"要达到什么"，后者是"不能超过什么"
4. **BBS 把测量权从 Agent 分离出来**，解决了传统 Agent 工作流中"自己评价自己"的根本缺陷
5. **收敛性依赖于反馈质量**，Validator 的设计重点不仅是正确性验证，更是信息量最大化的偏差测量

一句话总结：

`BBS 闭环不只是给 Agent 加了一把锁，而是给 Agent 加了一个眼睛——一个它自己无法欺骗的眼睛。`

## 17. 最后结论

我的结论很明确：

1. BBS 最适合做 `Agent 高风险动作的硬边界控制器`
2. 它最自然的第一批应用是：
   - 支付
   - API 调用授权
   - 企业自动审批
   - 自动化部署和执行权限限制
3. 它不适合直接解决：
   - 代码质量
   - 业务真实性
   - 软评价
4. 大多数落地场景第一阶段都不需要区块链
5. 如果真的要上链，也应该在闭环已经成立之后再讨论
6. 从控制论角度看，BBS 闭环不仅是安全机制，更是一个完整的负反馈控制系统——签名验证既是安全门禁，也是驱动 Agent 迭代收敛的反馈通道
7. 任务目标和安全约束在控制论框架下是同构的，同一套闭环既能约束"不能做什么"，也能驱动"必须做到什么"

最简洁的总结就是：

`BBS 不是为了让 Agent 更聪明，而是为了让 Agent 即使不够聪明，也出不了预设的硬边界——同时，当 Agent 足够聪明时，BBS 闭环会成为驱动它持续改进的控制回路。`
