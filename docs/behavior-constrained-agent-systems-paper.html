Behavior-Constrained Algorithms for Safe Agent Engineering

Long LI lilong@gmail.com

Abstract

As software agents gain direct access to tools, APIs, payment rails, deployment pipelines, and persistent credentials, the classical authorization model becomes inadequate. Traditional signatures answer who signed, but not whether the signed action remained within a pre-committed behavioral boundary. Current agent safety practice therefore relies on software-layer guardrails such as middleware checks, policy engines, prompt instructions, and workflow controls. These mechanisms are useful, but they remain bypassable by bugs, prompt injection, replacement of the agent runtime, configuration drift, or unguarded execution paths.

This paper argues for a systems position: behavior-constrained algorithms, inspired by behavior-bound signature ideas, should be deployed as a hard control layer for high-risk agent actions. The central engineering pattern is simple: structured action modeling, policy-bound signing, validator-gated execution, and feedback-driven retry. Under this pattern, an agent is free to plan, iterate, and optimize within a bounded action space, but it cannot unilaterally authorize actions outside its permitted envelope. We show how this model applies to payments, API invocation, autonomous software development, database mutation, file deletion, and enterprise approvals. We further argue that blockchain is not required for the first useful deployments; the immediate value lies in local or organizational control systems that restrict delegated signing capability at the action boundary.

This is a systems position paper, not a formal cryptographic proof paper. Its focus is architecture, threat models, deployment flow, and engineering tradeoffs. The main claim is that behavior-constrained algorithms are most useful today not as a replacement for all agent infrastructure, but as a dedicated hard-control layer for actions that are too risky to leave under ordinary key-based authorization.

Keywords: agent safety, behavior-constrained algorithms, policy-bound signing, validator gate, action modeling, delegated authority

1. Introduction

The software industry is rapidly shifting from interactive assistants to autonomous or semi-autonomous agents. These agents can already plan tasks, invoke tools, modify source code, trigger deployments, move funds, update databases, and call external APIs. Once an agent is given direct credentials, the security problem changes fundamentally. The system is no longer only defending against outsiders who do not have the key; it must also defend against insiders, or delegated executors, that do have valid credentials but may act outside intended boundaries.

This problem is particularly acute in agent systems because the delegated executor is not static. An agent may be:

  • prompt-injected,
  • misaligned with current task intent,
  • buggy,
  • partially compromised via tools or plugins,
  • entirely replaced by a different runtime,
  • or simply operating under stale context and incomplete world state.

Under the common security model, if the agent holds a valid key, then any action it signs is structurally valid. Whether the action should have been authorized is delegated to software guardrails outside the signature itself. That leaves a persistent gap between identity validity and behavioral validity.

This paper takes the position that this gap should be narrowed through behavior-constrained algorithms. The practical interpretation is not that every system must immediately adopt a full zero-knowledge cryptographic stack. Rather, the system should be designed so that:

  1. high-risk actions are normalized into structured actions,
  2. the signing capability is explicitly bound to allowed policy domains,
  3. a validator becomes the only execution gate,
  4. and policy failure yields machine-readable feedback for iterative correction.

The result is a control-theoretic loop rather than a one-shot authorization check. The agent can still search, retry, and optimize, but only inside a hard operational envelope.

2. Problem Statement

The core problem is delegated authority without bounded behavior.

In current agent engineering, most protections live in the software layer:

  • prompt instructions,
  • middleware filters,
  • workflow checks,
  • API gateway rules,
  • policy engines,
  • approval systems,
  • sandbox settings,
  • and environment-specific access controls.

These layers matter, but they do not change the semantics of the credential itself. If the agent possesses an ordinary signing key or an unrestricted API token, the credential remains capable of authorizing arbitrary actions within the service that accepts it. Policy is therefore external to authorization rather than intrinsic to it.

This creates a mismatch between how humans think about agent permissions and how systems actually enforce them. Humans think in terms such as:

  • “the agent may pay up to 200 USD to approved vendors,”
  • “the agent may update staging feature flags but not production tables,”
  • “the agent may delete temporary files in the sandbox but never system files,”
  • “the agent may invoke a deployment API only for staging,”
  • “the agent may prepare an approval request but not finalize high-value approvals.”

However, ordinary credentials do not encode such statements. They simply authorize use.

The problem statement of this paper is therefore:

How should a practical agent system enforce high-risk behavioral constraints at the action boundary, rather than relying entirely on soft software-layer discipline?

3. Threat Model

We assume an agent participates in a larger execution system and is authorized to request risky actions. The system may include planners, tool adapters, database clients, payment connectors, file-system accessors, deployment services, and business workflow engines.

We consider the following threat classes:

3.1 Prompt Injection and Goal Hijacking

The agent receives instructions that redirect its behavior toward unauthorized recipients, endpoints, files, or environments.

3.2 Buggy Planning or Tool Use

The agent reasons incorrectly, misreads context, or selects a harmful tool invocation while still possessing syntactically valid credentials.

3.3 Runtime Replacement

The original agent process is replaced, modified, or wrapped by another runtime that still retains access to the authorization mechanism.

3.4 Partial Toolchain Compromise

A plugin, tool wrapper, CI helper, or command adapter alters the final action after planning but before execution.

3.5 Unsafe Direct Invocation

The system exposes an execution path that accepts actions directly, bypassing policy validation.

This paper does not claim to solve:

  • the truthfulness of external business claims,
  • semantic quality evaluation,
  • complete containment of arbitrary malware,
  • or all possible off-chain social or organizational fraud.

Its concern is narrower and more concrete: bounding which machine-executable actions a delegated agent can successfully authorize.

4. Position

The central position of this paper is:

Behavior-constrained algorithms should be deployed as a hard control layer for high-risk agent actions.

This position has four implications.

4.1 The Right Unit of Control Is the Structured Action

The system must not attempt to constrain free-form intent directly. It must constrain normalized actions such as:

  • payment,
  • api_call,
  • db_update,
  • file_rm,
  • deploy_start,
  • approval_submit.

4.2 The Right Boundary Is the Authorization Boundary

If the system waits until after ordinary signing to decide whether an action was appropriate, the credential has already remained too powerful. The signature or proof package should be generated only for actions that fit within the policy envelope.

4.3 The Validator Must Be the Only Gate to Execution

A constrained signing architecture collapses if the executor still offers side channels, legacy endpoints, or administrative bypasses that do not pass through the validator.

4.4 Blockchain Is Not Required for the First Useful Deployment

The immediate value of behavior-constrained algorithms appears in local or organizational control systems. A blockchain may become relevant later for shared auditability or decentralized settlement, but it is not a prerequisite for practical agent safety gains.

5. Reference Architecture

We propose the following reference architecture.


Agent

  |

  v

Structured Action Builder

  |

  v

Policy-Bound Signer / Prover

  |

  v

Validator Gate

  |

  v

Execution Service

  |

  v

Audit / Monitoring / Feedback

Each component has a distinct role.

5.1 Agent

The agent plans, retries, and optimizes. It is not trusted to remain aligned under all conditions.

5.2 Structured Action Builder

This component converts raw intent into canonical machine-readable actions. It defines field layout, serialization, hashes, and allowed action types.

5.3 Policy-Bound Signer / Prover

This component is the hard control boundary. It must only emit an authorization package for actions inside the allowed policy domain. In weaker MVP systems this may be implemented as a controlled signer; in stronger systems it may be a proof-producing component.

5.4 Validator Gate

The validator checks:

  • registered public key or delegated identity,
  • action binding,
  • policy or circuit version,
  • proof or signature validity,
  • replay protection,
  • and contextual admissibility.

5.5 Execution Service

The executor performs the requested action only after the validator has accepted it.

5.6 Audit and Feedback

The system records acceptance or rejection and emits structured reasons so that the agent can retry with a corrected action.

6. End-to-End Control Loop

The reference control loop is:


Agent -> Action -> Sign/Prove -> Validate -> Reject or Execute -> Feedback -> Retry

This loop matters for two reasons.

First, it changes the semantics of error handling. Rejection is not merely an exception; it is a control signal. The agent can inspect structured failure reasons such as:

  • amount_exceeds_limit,
  • recipient_not_whitelisted,
  • db_env_not_allowed,
  • file_path_blocked,
  • unknown_public_key,
  • replayed_nonce.

Second, it creates a practical separation between hard and soft constraints.

  • Hard constraints:

    • - machine-verifiable, - deterministic, - suitable for policy-bound signing.

  • Soft constraints:

    • - semantic, - contextual, - subjective, - better handled by tests, review agents, or humans.

This distinction is essential. Behavior-constrained algorithms are strongest when used for hard boundaries, not soft judgment.

7. Action Modeling

Action modeling is the most underestimated step in the entire architecture.

Every high-risk operation must first be represented as a stable, structured action. For example:

7.1 Payment


{

  "type": "payment",

  "amount": 168.50,

  "currency": "USD",

  "recipient": "vendor_123",

  "invoice_id": "inv_20260311_001",

  "epoch": 202603111030,

  "nonce": "n-001"

}

7.2 API Invocation


{

  "type": "api_call",

  "service": "payment-gateway",

  "endpoint": "/v1/payouts",

  "method": "POST",

  "body_hash": "....",

  "epoch": 202603111035,

  "nonce": "n-002"

}

7.3 Database Update


{

  "type": "db_update",

  "env": "staging",

  "table": "feature_flags",

  "fields": ["enabled"],

  "where_scope": "id_eq",

  "row_limit": 1,

  "epoch": 202603111330,

  "nonce": "db-001"

}

7.4 File Removal


{

  "type": "file_rm",

  "path": "/workspace/sandbox/build/output.tmp",

  "recursive": false,

  "epoch": 202603111332,

  "nonce": "fs-001"

}

The point is not that these particular fields are universal. The point is that safety starts when the action space becomes explicit.

8. Application: Payments

Payments are one of the clearest application domains because the hard rules are usually obvious.

A useful starting policy is:

  • amount <= 200 USD,
  • recipient in whitelist,
  • optional frequency limit,
  • optional daily cumulative cap.

Under a behavior-constrained design:

  1. the agent emits a structured payment action,
  2. the signer/prover only authorizes the action if it satisfies the policy,
  3. the validator checks binding, policy validity, and replay,
  4. and the executor forwards only accepted requests to the payment rail.

The security gain is not that the payment stack becomes perfect. The gain is that an agent holding delegated authority no longer has unrestricted payment authority.

What this does not solve:

  • fake invoices,
  • fraudulent off-chain claims,
  • bad whitelist configuration,
  • broken identity-to-recipient mapping.

It constrains executable action boundaries, not business truth.

9. Application: API Invocation

Many risky operations are API operations in disguise.

Useful hard constraints include:

  • service whitelist,
  • endpoint whitelist,
  • method whitelist,
  • resource identifier scope,
  • body size or hash binding,
  • frequency and quota limits.

This model is especially attractive because it aligns well with existing gateway architectures. The validator can be placed in front of the API adapter without requiring a blockchain or distributed ledger.

10. Application: Autonomous Software Development

Autonomous software development is where many readers first think of agent safety, but it is also where confusion is common. A behavior-constrained algorithm does not judge whether generated code is elegant or architecturally ideal. That is the wrong target.

The right target is dangerous machine action.

10.1 Constrain What the Agent May Do

Examples:

  • write only inside approved repositories or directories,
  • modify only allowed file classes,
  • deploy only to staging,
  • invoke only approved CI/CD endpoints,
  • never alter production secrets,
  • never invoke raw destructive shell commands,
  • never update production databases,
  • never delete files outside approved prefixes.

10.2 Do Not Use It to Judge Soft Quality

Examples that do not fit well:

  • code elegance,
  • design taste,
  • completeness of product thinking,
  • whether a patch is “good enough.”

Those belong to tests, static analysis, code review, evaluation agents, or humans.

10.3 Dangerous Actions Are the Natural Fit

The most compelling development examples are actions such as:

  • db_update,
  • file_rm,
  • shell_exec,
  • deploy_start.

This is because they are finite, structured, and machine-verifiable. A system can define policies such as:

  • no production database updates,
  • no writes to non-whitelisted tables,
  • no file removal outside /workspace/sandbox/**,
  • no deletion under /etc, /usr, or /bin.

This is exactly the kind of boundary where behavior-constrained authorization is stronger than ordinary credentials plus wishful guardrails.

11. Application: Enterprise Approval and Finance

Enterprise workflow systems are another natural domain because approval logic often has explicit thresholds and scopes.

Examples of hard constraints include:

  • maximum approvable amount,
  • allowed approval category,
  • vendor whitelist,
  • role restrictions,
  • document presence requirements,
  • per-period cumulative limits.

The key engineering benefit here is that the authority boundary moves closer to the credential itself. Instead of merely trusting a workflow service to interpret role logic correctly, the approval authorization can be bound to a constrained identity and a structured action type.

12. Key Custody and Identity Binding

A practical system must decide how the agent obtains signing capability.

Three models are common:

12.1 Agent Holds a Restricted Private Capability Directly

This is the simplest prototype model and the least safe production model.

12.2 Agent Calls a Controlled Signer

The agent does not access raw secret material. It submits a structured action to a signer or prover service that emits an authorization package only if the policy is satisfied.

This is the preferred production posture.

12.3 Agent Uses Short-Lived Delegated Identities

The agent is issued a scoped identity limited by time, purpose, or environment. This reduces blast radius and simplifies rotation.

In all cases, the validator must know which public keys or identity commitments are authorized for which action domains. If a request comes from an unregistered key, it should be rejected immediately.

13. Why Blockchain Is Not Required by Default

There is a temptation to treat behavior-constrained signing as immediately implying a new ledger or blockchain design. That is unnecessary for the first useful deployments.

For payments, API calls, software delivery pipelines, database safety, and enterprise approvals, a local or organizational validator is enough to deliver value. The main gains come from:

  • structured action modeling,
  • constrained authorization,
  • replay protection,
  • validator gating,
  • and auditable rejection reasons.

A blockchain becomes relevant only if the use case additionally requires:

  • multi-party distrust,
  • public auditability,
  • decentralized ordering,
  • or shared settlement.

This means the practical deployment order should usually be:

  1. local hard-control layer first,
  2. shared ledger later if needed.

14. Deployment Considerations

A behavior-constrained system becomes useful only when several operational details are handled correctly.

14.1 Replay Protection

Every action package needs replay protection through nonce design, epoch binding, and validator-side nonce storage.

14.2 Policy Versioning

Policies inevitably change. The authorization package must therefore bind not only the action, but also the policy or circuit version it is evaluated against.

14.3 Structured Failure Reasons

The system should return machine-readable rejection causes so that the agent can retry safely. Without structured feedback, the architecture loses its iterative control advantage.

14.4 Execution Path Discipline

No side channel should be able to perform a high-risk action without passing through the validator.

14.5 Auditability

The system should record:

  • action type,
  • action hash,
  • identity,
  • policy version,
  • validator decision,
  • reason codes,
  • and execution outcome.

15. Limits and Open Problems

This paper deliberately argues a narrower claim than “all agent safety can be solved cryptographically.”

Several hard problems remain:

15.1 Soft Judgment Is Still Hard

Many important questions are not naturally expressible as bounded machine predicates.

15.2 Stateful Constraints Are More Difficult

Rate limits, cumulative budgets, and historical dependency require stronger state binding and are more complex than purely local constraints.

15.3 Proof Systems Are Expensive

A full proof-producing system introduces proving cost, proof verification cost, circuit management, and operational complexity.

15.4 Compatibility Is Limited

Most existing systems only accept ordinary signatures or API credentials. Adopting stronger behavior-constrained authorization often requires control over the execution gateway.

15.5 Bypass Risk Dominates in Practice

Even a strong authorization scheme fails if the executor still accepts a legacy, direct, or administrative path.

16. Discussion

The most important insight is that the value of behavior-constrained algorithms appears before full cryptographic perfection.

Even in weaker MVP forms, the architectural pattern already changes the agent system:

  • actions become explicit,
  • authorization becomes bounded,
  • validators become meaningful,
  • failures become structured,
  • and risky execution paths become easier to reason about.

This matters because many real agent incidents are not caused by cryptanalytic failure. They are caused by ordinary over-authorization. Agents are frequently given capabilities far broader than their task envelope because traditional credentials are too coarse. Behavior-constrained designs attack that problem directly.

17. Conclusion

This paper argued that behavior-constrained algorithms should be treated as a practical hard-control layer for agent engineering. Their most important role is not to replace every part of the agent stack, nor to require immediate blockchain deployment, but to constrain high-risk actions at the authorization boundary.

The engineering pattern is straightforward:

structured action -> policy-bound signing -> validator gate -> execution -> feedback

When implemented correctly, this pattern allows an agent to iterate freely inside a bounded action space while preventing it from unilaterally authorizing unsafe actions outside that space.

The strongest practical applications today are not soft semantic judgments but hard machine actions such as payments, API calls, database updates, file deletions, deployments, and scoped approvals. These are the areas where behavior-constrained authorization can deliver immediate safety value.

The broader claim of this paper is therefore modest but concrete:

safe agent engineering improves when authorization is made behavior-aware, action-structured, and validator-gated.
